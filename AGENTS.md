# AGENTS.md — Sake Tasting Log App 작업 지침

이 파일은 AI 에이전트/코딩 에이전트가 이 프로젝트를 수정할 때 반드시 따라야 할 기준이다.

현재 프로젝트의 기준 문서는 `PROJECT_SAKE_REVISED.md`이다. 기존 `PROJECT.md`보다 이 문서를 우선한다.

---

## 1. 현재 제품 방향

이 프로젝트는 당장 범용 주류 기록 앱이 아니다.

현재 목표는 다음 하나다.

> 사케를 마실 때 빠르게 사진, 기본 정보, 맛의 방향, 다시 마실 의향, 장소와 동행 정보를 기록하는 앱을 만든다.

위스키, 진, 럼, 와인, 한국 전통주 등은 지금 구현하지 않는다. 다만 나중에 확장할 수 있도록 `drink_type` 필드 정도만 남긴다.

---

## 2. 최우선 원칙

1. 사케 MVP를 먼저 완성한다.
2. 사용자가 술을 마신 상태에서도 입력할 수 있게 단순해야 한다.
3. 술 이름만 필수다. 나머지는 선택 입력이다.
4. 사진 여러 장을 저장할 수 있어야 한다.
5. 커스텀 태그는 추가 후 다음 기록에서도 다시 보여야 한다.
6. UI 순서는 `PROJECT_SAKE_REVISED.md`의 “오늘의 한 잔 기록” 순서를 따른다.
7. 확장성 때문에 현재 UX를 복잡하게 만들지 않는다.

---

## 3. 하지 말 것

다음 작업은 사용자가 명시적으로 요청하기 전까지 하지 않는다.

- 위스키/진/럼/와인 전용 화면 구현
- Nose / Palate / Finish 구조 강제
- 별점 시스템 추가
- 복잡한 통계 대시보드 구현
- AI 추천 구현
- OCR 구현
- 라벨 자동 인식 구현
- 공개 공유 기능 구현
- 동행/안주를 별도 테이블로 과도하게 정규화
- 가격/도수/용량을 처음부터 숫자 정규화

## 5. 필드 정의

### 5.1 사진

- 기록 하나에 사진 여러 장을 연결한다.
- 첫 번째 사진을 대표 이미지로 사용한다.
- 사진이 없더라도 기록 저장은 가능해야 한다.

### 5.2 기본 정보

필드:

- 술 이름: 필수
- 지역: 선택
- 양조장: 선택
- 쌀: 선택
- 종류: 선택
- 일본주도: 선택
- 도수: 선택
- 용량: 선택
- 가격: 선택

초기 구현에서는 일본주도, 도수, 용량, 가격을 문자열로 저장한다.

예:

- `+1`
- `15%`
- `720ml`
- `3000 yen`

### 5.3 다시 마실까?

하나만 선택한다.

표시값:

- 별로
- 잘모르겠음
- 다시 마신다

저장값:

- `no`
- `unsure`
- `yes`

선택하지 않아도 저장 가능하게 둔다.

### 5.4 평가

모든 평가 항목은 단일 선택이다.

#### 달콤함 - 드라이함

표시값:

- 아주 달콤함
- 달콤함
- 보통
- 드라이함
- 아주 드라이함

저장값:

- `1`: 아주 달콤함
- `2`: 달콤함
- `3`: 보통
- `4`: 드라이함
- `5`: 아주 드라이함

#### 은은함 - 화려함

표시값:

- 은은한향
- 보통
- 화려한향

저장값:

- `1`: 은은한향
- `2`: 보통
- `3`: 화려한향

#### 산미

표시값:

- 산미없음
- 산미보통
- 산미높음

저장값:

- `1`: 산미없음
- `2`: 산미보통
- `3`: 산미높음

#### 깔끔함 - 감칠맛

표시값:

- 깔끔함
- 보통
- 감칠맛좋은

저장값:

- `1`: 깔끔함
- `2`: 보통
- `3`: 감칠맛좋은

### 5.5 한줄 메모

자유 입력이다.

권장 placeholder:

```text
오늘 마신 느낌을 한 줄로 남겨보세요.
```

### 5.6 특성 태그

태그 그룹은 세 가지다.

- 맛 태그: `taste`
- 향 태그: `aroma`
- 느낌 태그: `mood`

각 그룹은 여러 개 선택 가능하다.

각 그룹 끝에는 `+` 버튼을 둔다.

사용자가 `+`로 태그를 추가하면:

1. 앞뒤 공백을 제거한다.
2. 빈 문자열은 저장하지 않는다.
3. 같은 그룹 안에 같은 label이 있으면 새로 만들지 않는다.
4. 새로 추가된 태그는 자동 선택한다.
5. 다음 기록에서도 같은 그룹 안에 표시한다.
6. 길이는 20자 이하로 제한한다.

기본 태그는 다음과 같다.

맛 태그:

- 산뜻
- 감칠
- 부드러움
- 진함
- 깔끔함
- 쌉쌀함
- 달콤함

향 태그:

- 누룩
- 쌀
- 과일
- 꽃
- 사과
- 파인애플
- 멜론
- 배
- 허브
- 요구르트
- 견과

느낌 태그:

- 독특
- 무난
- 안주랑 먹기 좋음
- 식전주

주의: `부드러음`이 아니라 `부드러움`이다.

### 5.7 외부 정보

필드:

- 장소: 선택
- 날짜: 기본값 오늘
- 동행: 선택
- 안주: 선택

날짜는 새 기록 생성 시 오늘 날짜로 자동 입력한다. 사용자가 수정할 수 있어야 한다.

## 6. 데이터 모델 기준

기본 테이블은 다음 구조를 따른다.

- `users`
- `sake_records`
- `sake_images`
- `tags`
- `record_tags`

### 6.1 sake_records

필수 필드:

- `id`
- `owner_id`
- `drink_type`
- `name`
- `consumed_date`
- `created_at`
- `updated_at`

`drink_type`의 기본값은 `sake`이다.

`sake_records.name`은 비어 있으면 안 된다.

초기 버전에서 아래 필드는 `TEXT`로 둔다.

- `sake_meter_value`
- `abv`
- `volume`
- `price`
- `companions`
- `food_pairing`

### 6.2 sake_images

- `record_id`로 사케 기록과 연결한다.
- 삭제 시 기록에 연결된 이미지는 함께 삭제되어야 한다.
- R2 경로 권장 형식:

```text
images/{owner_id}/sake/{record_id}/{image_id}.jpg
```

### 6.3 tags

- 기본 태그는 `owner_id = NULL`로 둔다.
- 사용자 추가 태그는 `owner_id`를 가진다.
- `tag_group`은 `taste`, `aroma`, `mood` 중 하나다.
- `drink_type`은 기본적으로 `sake`다.

### 6.4 record_tags

- 하나의 기록에 여러 태그를 연결한다.
- 같은 기록에 같은 태그가 중복 연결되면 안 된다.

## 7. UI 구현 기준

### 7.1 화면 밀도

입력 화면은 빽빽하면 안 된다.

사케 정보는 많지만 사용자가 전부 입력하지 않아도 된다는 느낌을 줘야 한다.

### 7.2 버튼형 선택

평가 항목은 슬라이더보다 버튼형 선택을 우선한다.

이유:

- 모바일에서 누르기 쉽다.
- 의미가 명확하다.
- 사용자가 초보여도 고를 수 있다.

### 7.3 다시 마실까 위치

`다시 마실까?`는 평가보다 위에 둔다.

이 값은 이 앱의 핵심 판단값이다.

### 7.4 태그 UI

- 칩 형태로 보여준다.
- 선택된 태그는 명확하게 강조한다.
- `+` 버튼은 각 태그 그룹 끝에 둔다.
- 새 태그 추가 후 자동 선택한다.

### 7.5 리스트 화면

리스트에는 최소한 다음을 보여준다.

- 대표 사진
- 술 이름
- 종류 또는 지역
- 다시 마실까?
- 날짜
- 주요 태그 2~3개

기본 정렬은 최신순이다.

### 7.6 상세 화면

상세 화면은 다음 순서를 따른다.

1. 사진 갤러리
2. 술 이름
3. 다시 마실까?
4. 평가 요약
5. 한줄 메모
6. 특성 태그
7. 기본 정보 전체
8. 외부 정보
9. 수정 버튼

## 8. API 기준

최소 API는 다음과 같다.

```http
GET /api/sake-records
POST /api/sake-records
GET /api/sake-records/:id
PUT /api/sake-records/:id
DELETE /api/sake-records/:id

POST /api/sake-records/:id/images
DELETE /api/sake-records/:id/images/:imageId

GET /api/tags?drink_type=sake
POST /api/tags

GET /api/sake-records/search?q=...
```

API 구현 시 모든 read/write는 현재 사용자 기준으로 필터링한다.

다른 사용자의 기록에 접근하려 하면 `403`을 반환한다.

## 9. 인증/권한 기준

Cloudflare D1/R2를 사용할 경우 인증은 필수다.

- Google OAuth를 사용한다.
- Google `sub` 값을 `provider_user_id`로 저장한다.
- 이메일을 고유 식별자로 쓰지 않는다.
- 세션 쿠키를 사용한다.
- 모든 사용자 소유 데이터는 `owner_id`를 가진다.
- 이미지 업로드도 인증된 API를 통해서만 처리한다.

로컬 전용 프로토타입을 만드는 경우에도 나중에 `owner_id`를 붙일 수 있게 구조를 막지 않는다.

## 10. 코드 작업 규칙

에이전트는 작업 전 다음을 확인한다.

1. 현재 요청이 사케 MVP에 필요한가?
2. `PROJECT_SAKE_REVISED.md`와 충돌하지 않는가?
3. 사용자가 명시하지 않은 확장 기능을 넣고 있지는 않은가?
4. 입력 흐름을 복잡하게 만들고 있지는 않은가?

코드 수정 시 다음을 지킨다.

- 작은 단위로 수정한다.
- 기존 동작을 깨지 않는다.
- 타입이 있다면 명확히 정의한다.
- 저장값과 표시값을 분리한다.
- 하드코딩이 필요한 기본 태그/선택지는 한 곳에 모은다.
- 사용자가 입력한 문자열은 trim 처리한다.
- 필수값 검증은 `name` 중심으로 한다.
- 날짜 기본값은 오늘 날짜로 둔다.

## 13. 완료 기준

MVP 완료로 볼 수 있는 최소 조건:

- 새 사케 기록을 만들 수 있다.
- 술 이름이 필수 검증된다.
- 사진 여러 장을 추가할 수 있다.
- 기본 정보를 저장할 수 있다.
- 다시 마실까 값을 저장할 수 있다.
- 4개 평가값을 저장할 수 있다.
- 한줄 메모를 저장할 수 있다.
- 기본 태그를 선택할 수 있다.
- 커스텀 태그를 추가할 수 있다.
- 추가한 태그가 다음 기록에도 보인다.
- 날짜 기본값이 오늘로 들어간다.
- 리스트에서 기록을 볼 수 있다.
- 상세 화면에서 전체 기록을 볼 수 있다.
- 기록을 수정할 수 있다.
- 기록을 삭제할 수 있다.

## 14. 최종 판단 기준

기능을 추가할지 말지 애매하면 다음 질문으로 판단한다.

> 이 기능이 5월 일본 여행 중 사케를 빠르게 기록하는 데 직접 도움이 되는가?

답이 아니면 지금은 하지 않는다.

이 프로젝트는 작고 정확하게 시작해야 한다. 범용성과 똑똑함보다 먼저 필요한 것은 기록이 실제로 쌓이는 것이다.
